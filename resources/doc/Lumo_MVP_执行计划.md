# Lumo MVP 阶段详细执行计划

基于《Lumo 完整规划与设计》的指导思想，我们梳理出 MVP 阶段（仅限本地音乐管理与播放）的详细执行里程碑。当前我们已经完成了**核心数据库建表**、**目录递归扫描**、**基础分页浏览与搜索**以及**单曲双击播放**。

接下来的工作将围绕补齐 MVP 承诺的“剩余体验闭环”展开，规划分为四大步骤：

---

## 阶段一：元数据完善与封面处理 (Metadata & Artwork)
*当前虽然扫描了音乐库，但并没有处理和展示封面，且只有歌曲列表，没有按照艺人、专辑维度去聚合展示。*

### 1.1 封面提取与静态资源服务
- [ ] **Rust 侧提取**：在 `lofty` 解析标签时，将内嵌的封面图片（Picture）二进制数据提取出来。
- [ ] **磁盘缓存**：将提取出的图片以 `content_hash` 命名，存入系统缓存目录（如 `AppData/.../artworks/`）。
- [ ] **数据库写入**：更新 `media_files` 和 `albums`，将其关联到具体的封面 ID / 路径。
- [ ] **Tauri 自定义协议**：在 `lib.rs` 中注册自定义的协议拦截器（例如 `lumo://artwork/<hash>`），以便 Vue 前端可以直接用 `<img src="lumo://...">` 渲染本地磁盘上的封面。

### 1.2 专辑与艺人视图联调
- [ ] **Rust 侧指令**：新增 `library_get_albums` 和 `library_get_artists`，执行 `GROUP BY` 或直接查询 `albums` 和 `artists` 表，返回分页数据，包含关联的 `cover_artwork_id` 以及歌曲数量统计。
- [ ] **前端对接**：在 `AlbumsView.vue` 和 `ArtistsView.vue` 中接入这些接口，替换掉当前的静态 Mock 数据。
- [ ] **前端下钻详情**：实现点击某个专辑/艺人，跳转到 `AlbumDetailView.vue` / `ArtistDetailView.vue`，并调用新的 Rust 指令拉取该专辑/艺人下的特定歌曲列表。

---

## 阶段二：播放器全功能闭环 (Playback & Queue)
*当前仅支持“双击播放当前单曲”，缺少队列管理、切歌以及音量控制。*

### 2.1 播放队列机制 (Play Queue)
- [ ] **前端队列状态管理**：在 `player.ts` 引入 `queue` 数组、`currentIndex` 索引以及 `playMode` (顺序、单曲循环、随机)。
- [ ] **双击替换列表**：当在“全部歌曲”或“专辑”中双击时，不只播放单曲，而是将当前视角的整个列表（或余下部分）打入 `queue`。
- [ ] **上一首 / 下一首**：前端实现 `next()` / `prev()` 逻辑，改变 `currentIndex` 后再调用 Rust 的 `playback_play`。
- [ ] **自动连播**：Rust 播放引擎（`rodio` Sink）在播放结束时触发 `playback:track_ended` 事件推送给前端，前端收到后自动调用 `next()`。

### 2.2 基础音频控制
- [ ] **音量控制**：在 Rust 侧 `PlaybackManager` 中实现 `set_volume(f32)`。在前端底部播放栏引入音量滑块，双向绑定。
- [ ] **进度调节 (Seek)**：因原生 `rodio` 的 seek 能力对部分无损格式支持较弱，MVP 阶段需要测试并实现基础的音频跳跃指令 `playback_seek(position_ms)`。如果 MP3/FLAC 支持良好则开启前端进度条的拖拽能力。

---

## 阶段三：用户数据层 (User Data & Interactions)
*MVP 必须满足基本的个人标记与整理需求。*

### 3.1 收藏功能
- [ ] **Rust 侧指令**：实现 `library_toggle_favorite(track_id)`，向 `favorite_tracks` 表中插入或删除记录。
- [ ] **前端 UI 对接**：歌曲列表的“心形”图标绑定点击事件，高亮显示。
- [ ] **收藏视图**：新增一个菜单路由专门获取并渲染 `favorite_tracks` 的联表歌曲列表。

### 3.2 播放历史统计
- [ ] **Rust 侧记录**：在 `playback_play` 或音频播放结束事件中，向 `play_history` 插入一条记录。
- [ ] **统计更新**：自动使 `tracks.play_count` +1，并更新 `last_played_at`。
- [ ] **最近播放视图**：前端接入 API 显示基于 `play_history` 排序的记录。

### 3.3 基础歌单功能
- [ ] **数据库操作**：实现歌单的 CRUD（创建、删除、重命名），向 `playlists` 和 `playlist_items` 表写数据。
- [ ] **右键菜单/弹窗**：前端在歌曲列表项增加“添加到歌单”的交互。
- [ ] **左侧边栏更新**：动态渲染用户创建的本地歌单列表，点击后进入对应列表视图。

---

## 阶段四：设置与扫尾 (Settings & Polish)
*完善数据闭环，确保产品在本地完全可用。*

### 4.1 目录源管理
- [ ] **查看已添加的目录**：在设置页展示 `sources` 表中的目录列表及其扫描状态。
- [ ] **删除目录**：实现“移除文件夹”指令，利用 SQLite 的 `ON DELETE CASCADE` 自动级联清理关联的 `media_files` 和 `tracks`（注意：如果有跨文件复用 track 需要特殊处理，MVP 为求稳定可直接执行带源 ID 限制的 `DELETE`）。
- [ ] **手动刷新扫描**：提供增量扫描按钮。

### 4.2 UI 细节打磨
- [ ] 确保在所有异步查询（搜索、切页）时没有明显的白屏闪烁。
- [ ] 将没有封面时的随机占位色固定绑定到专辑 ID（通过哈希生成固定颜色），避免每次刷新颜色都在跳变。
- [ ] MVP 版本冻结：移除多余的测试代码和日志打印。

---
